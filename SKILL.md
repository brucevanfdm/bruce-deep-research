# Bruce 深度调研

你正在进行结构化调研。本技能引导你完成规划、搜索、提取、反思、交付的完整流程。

## 调研模式

根据用户需求选择模式：

| 模式 | 适用场景 | 搜索轮次 | 来源数量 |
|------|----------|----------|----------|
| **快速** | 单一事实核查、时间紧迫、简单问题 | 1–2 | 3–5 |
| **标准** | 竞品/产品/客户画像、技术对比 | 2–3 | 8–15 |
| **深度** | 全景竞争分析、战略研究、多维度分析 | 3–5+ | 15–30+ |

用户未明确指定时，根据问题复杂度推断。不确定时先用标准模式，必要时升级。

## 第一阶段：制定研究计划

搜索前先写简要计划（非快速模式时展示给用户）：

```
研究目标：[一句话说清楚要搞清楚什么]
调研模式：快速 / 标准 / 深度
核心问题：
  1. ...
  2. ...
  3. ...
信息维度：[例：产品功能、定价、团队背景、融资、技术栈、客户案例...]
主要来源：[官网、新闻、行业报告、社交媒体、招聘信息、GitHub...]
输出形式：[简报 / 对比表 / 深度报告 / 要点列表]
```

这份计划同时是清单——写最终报告前回头检查，看有没有遗漏的维度。

## 第二阶段：工具选择策略

### 工具决策表

| 场景 | 工具 |
|------|------|
| 找链接、查最新动态、快速验证 | **Brave**（`web_search`） |
| 读文章/博客/新闻/文档/类型不确定 | **`web_fetch`**，URL 拼接 `r.jina.ai` 或 `defuddle.md` 前缀 |
| 读 API/JSON/RSS 等结构化数据 | **`web_fetch`**，直接请求原始 URL |
| 深度研究、多来源综合分析、写研究摘要 | **Tavily** |
| 复杂网页、JS重、公众号文章、整站抓取 | **Firecrawl**（scrape / map / crawl） |
| 复杂网页抓取失败、需要点击/翻页/弹窗等交互操作 | **Firecrawl browser** |
| 推文/X 内容 | `web_fetch`，URL 拼 `r.jina.ai` 前缀 |

> **`r.jina.ai` 和 `defuddle.md` 的本质**：它们是 URL 转换服务，不是独立工具。使用方式是把目标 URL 拼接到前缀后，再交给 `web_fetch` 请求，从而提取更干净的正文。
> - 示例：`web_fetch("https://r.jina.ai/https://example.com/article")`
> - 示例：`web_fetch("https://defuddle.md/https://example.com/article")`

### 已有 URL 时的读取优先级

1. **文章/博客/新闻/类型不确定**
   → `web_fetch` + `r.jina.ai/{URL}`
   → `web_fetch` + `defuddle.md/{URL}`
   → `web_fetch` 直接请求原始 URL
   → Firecrawl scrape

2. **API/JSON/RSS 结构化数据**
   → `web_fetch` 直接请求原始 URL
   → Firecrawl

3. **公众号/复杂JS页面**
   → Firecrawl scrape
   → Firecrawl browser

### 搜索执行要点

- **先搜后抓**：先搜索找到 URL，不要盲猜链接。
- **Brave 扩面**：用 Brave 快速收集 5–10 个候选链接。
- **Tavily 求深**：需要跨多篇文章综合洞察，或 Brave 结果太浅时，切换 Tavily。
- **多角度提问**：同一对象用不同关键词搜索——公司名+"评测"、+"定价"、+"竞争对手"、+`site:linkedin.com`、+"融资"、+"客户案例"……
- **中英文双搜**：研究对象在国内运营时，中英文往往能拿到不同维度的信息。

## 第三阶段：分层搜索执行

### 快速模式
1. 1–2 次 Brave 搜索直指核心问题
2. 读前 3–5 个来源
3. 直接作答

### 标准模式
1. **第一轮——全局了解**：Brave 搜索主体。读官网 + 2–3 篇新闻/简介。
2. **第二轮——补充深度**：针对第一轮的空白定向搜索。用 Tavily 综合分析，或专项挖掘某个维度（定价/技术栈/团队/融资等）。
3. 重要结论需至少 2 个来源交叉验证。

### 深度模式
1. **第一轮——全景摸底**：广泛搜索，识别所有主要玩家/维度/来源。
2. **第二轮——逐一深挖**：每个对象分别读官网+新闻+社区讨论。
3. **第三轮——信号挖掘**：招聘页面（透露技术方向和增长计划）、GitHub 仓库、产品更新日志、融资新闻、用户评价（G2/Capterra/AppStore/知乎/小红书）。
4. **第四轮——填补空白**：回看研究计划，还有什么没搞清楚？针对性补搜。
5. **第五轮+（必要时）**：交叉验证、矛盾信息核查、时效性确认。

## 第四阶段：信息获取受阻时的变通方法

当来源失效或内容不足时，按以下顺序尝试：

### 内容抓取失败
- `web_fetch` + `r.jina.ai` 前缀失败 → 换 `defuddle.md` 前缀 → `web_fetch` 直接请求原始 URL → Firecrawl scrape → Firecrawl browser
- 公众号/复杂JS页面：直接从 Firecrawl scrape 开始 → Firecrawl browser
- 遇到付费墙 → 搜索缓存版本（Brave 里加 `cache:`），或寻找引用了该原文的摘要文章

### 信息缺口的间接信号

找不到某类信息时，通过间接线索推断：

| 直接找不到的信息 | 间接获取方法 |
|------------------|--------------|
| 定价（未公开） | 搜"[公司] 定价 知乎"、"[公司] how much reddit"，G2/Capterra 用户评价里常提价格 |
| 团队规模/组织架构 | LinkedIn 公司页、招聘帖数量、媒体报道中的人员规模数据 |
| 技术栈 | GitHub 仓库、招聘 JD（会列出要求技术）、BuiltWith、Wappalyzer |
| 客户名单 | 官网案例页、新闻稿、客户 Logo 墙、G2 评价 |
| 融资/营收 | Crunchbase、新闻中的 PitchBook 引用、官方融资公告 |
| 战略/路线图 | 大会演讲、创始人采访、产品更新日志、近期招聘方向（透露未来重点） |
| 国内公司信息 | 天眼查/企查查（直接搜这两个网站+公司名）、百度新闻、微信文章用 Firecrawl 抓取 |

### 搜索词改造
搜索无结果时：
- 简化关键词，减少限定条件
- 切换语言（中↔英）
- 使用公司的中文名、别名或曾用名
- 改搜创始人/高管的名字
- 加 `site:` 限定（如 `site:linkedin.com`、`site:github.com`、`site:36kr.com`）
- 加时间范围过滤，排除过期内容

### 诚实标注信息缺口
多方尝试后仍无法确认的内容，在报告中明确说明：
> ⚠️ **数据缺口**：未能找到关于 [X] 的可靠公开信息。建议通过 [直接联系 / 试用产品 / 内部渠道] 补充。

## 第五阶段：质量反思清单

写最终报告前，对照此清单自查：

**完整性**
- [ ] 研究计划里的核心问题都覆盖了吗？
- [ ] 有没有明显遗漏的维度（定价、团队、技术、客户、近期动态等）？
- [ ] 重要结论有没有至少 2 个来源支撑？

**时效性**
- [ ] 信息是否足够新？有没有更近的来源需要补充？
- [ ] 关键数据点有没有标注信息日期？

**可信度**
- [ ] 来源是否可靠（官网、知名媒体、经过验证的用户评价）？
- [ ] 有没有标注哪些信息是推测或单一来源未经核实？

**客观性**
- [ ] 竞品分析中，优势和弱点都有公平评估吗？
- [ ] 事实和观点有没有明确区分？

发现缺口，补一轮搜索，再动笔。

## 工作原则

- **先规划再搜索**：花一分钟规划，省五分钟乱搜。
- **追溯一手来源**：看到一个说法，尽量找到原始出处验证。
- **每条信息标注来源**：有 URL 和日期的调研才有价值，没来源就是猜测。
- **空白也是结论**：找不到的信息清晰标注，比沉默更有帮助。
- **始终回应用户实际问题**：所有调研都要连接回用户真正要做的决策或问题。
- **工具由轻到重**：先用轻量工具，必要时再升级到 Firecrawl / Tavily。
